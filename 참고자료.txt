# 좋아 지금까지의 내용을 좀 정리해 줘. 바이브 코딩 하려고 해

## 🗒️ Answer

지금까지 논의를 **바이브 코딩 관점에서 실행 가능한 구조**로 정리하면 다음과 같습니다.

---

### 1. 핵심 개념

* **학생–AI 짝프로그래밍**

  * 학생 = Driver (주도적 코딩)
  * AI = Navigator (힌트, 설명, 리뷰)
* AI는 “답 제공자”가 아니라 **작업 수행형 에이전트**로 설계 [[3](https://www.gpters.org/research/post/0zBwFhaIKvC8R1N)]
* 대상은 대학교 1학년(처음 C언어를 배우는 학생) 
* 과제는 총 10개로 구성

---

### 2. 에이전트 목표

1. 사고 유도 (힌트 중심)
2. 코드 품질 향상
3. 학습 동기 및 몰입 강화
4. 협력적 문제 해결 경험 제공
5. 컴퓨팅 사고력 및 프로그래밍 자기효능감 향상

---

### 3. 핵심 기능

* 코드 생성(C언어) / 리팩토링
* 오류 탐지 + 원인 설명
* 단계적 힌트 제공
* 대안 비교 및 선택 지원
* 작업 흐름 이해 및 실행 (Agent 특징) [[1](https://bcho.tistory.com/1467)]

---

### 4. 화면 구조 (중요)

* 좌: 문
* 가운데: 코드 작성
* 우: AI 협력 패널(AI 챗봇)
  → “코드–AI–문제” 3축 구조

---

### 5. 인터랙션 설계

* 선택 기반: 적용 / 비교 / 무시
* 단계형 지원: 힌트 → 설명 → 코드
* 기록: 대화 로그 + 리플렉션

---

### 6. 바이브 코딩 적용 흐름

1. 문제 제시
2. 학생 코드 작성
3. AI 피드백 요청
4. 수정/개선 반복
5. 결과 공유 및 성찰

---

핵심은 단순 코딩 도구가 아니라
**“협력적 사고를 설계하는 AI 파트너 시스템”**으로 구현하는 것입니다.

# 여기서 교사는 대시보드는 어떻게?

## 🗒️ Answer

교사 대시보드는 **학생–AI 짝프로그래밍을 ‘오케스트레이션’하는 통제·지원 인터페이스**로 설계하는 것이 핵심입니다.

---

### 1. 핵심 목적

1. **학습 상태 파악** (진행률, 오류, 참여도)
2. **개입 판단 지원** (언제, 누구에게 개입할지)
3. **AI 활용 모니터링** (의존 vs 활용 균형)

→ AI는 교사를 대체하는 것이 아니라 **교사의 판단을 증강**하는 도구 [[3](https://link.springer.com/article/10.1007/s10639-024-12523-3)]

---

### 2. 필수 구성 요소

1. **학생 현황 패널**

   * 코드 진행률, 과제 완료율
   * 오류 발생 빈도 (디버깅 필요 학생 식별)

2. **AI 상호작용 로그**

   * 학생–AI 대화 내용
   * 힌트 요청 횟수 (과의존 탐지)

3. **알림/위험 신호**

   * 장시간 정체
   * 반복 오류
   * AI 답 복붙 패턴

4. **개입 도구**

   * 문제 제공
   * 힌트 직접 제공
   * 과제 난이도 조정
   * AI 개입 수준 조절 (강/중/약)

---

### 3. 고급 기능

* **학습 분석(Analytics)**: 성취도·참여도 시각화
* **AI 추천 개입**: “이 학생 개입 필요” 자동 제안
* **클래스 전체 패턴**: 공통 오류, 개념 취약점

→ AI 에이전트는 학습 데이터를 기반으로 교사 의사결정을 지원 ---

### 4. 설계 원칙

* **가시성**: 한 화면에서 전체 + 개별 상태 파악
* **개입 가능성**: 즉시 피드백 제공
* **균형 유지**: AI 자동화 vs 교사 통제

---

정리하면, 교사 대시보드는 단순 모니터링이 아니라
**“AI-학습-교사 간 상호작용을 조율하는 컨트롤 타워”**입니다.

# 후기 (2026-05-13)

이 메모는 프로젝트 시작 시점의 초안이다. 실제 구현은 [research.md](research.md) 의 12 주 로드맵 + 운영 이터레이션 25 회를 거쳐, **9 개 멀티 에이전트 · 11 개 과제 · 교사 7-layer 코크핏 · xAPI 분석 레이어 · 자동 배포 인프라**까지 확장됐다. 초안의 핵심 직관 (3 패널 · Navigator · 짝프로그래밍) 은 그대로 유지하되, 다음 영역이 본격적으로 구체화됐다.

- 원본의 **"AI 개입 수준 강/중/약"** → 실제 동작이 다른 **4-mode spectrum** (solo · pair · coach · exam) + HINT_CEILING + Safety 유사도 임계 차등
- 원본의 **"리플렉션 + 대화 로그"** → **xAPI events · turn-level conversations · KC mastery · misconceptions** + 결정적 BKT 갱신 cron
- 원본의 **"학습 분석"** → 교사 실시간 대시보드 + **Research Lab** (Paper 1 · 3 시각화: cascade · offloading · conversation arcs · linguistic profile)

상세 스펙은 [research.md](research.md) §5–§11, 운영 이력·플레이북은 [CLAUDE.md](CLAUDE.md) 의 *변경 이력 표* 와 *운영 플레이북* 섹션 참조.

# 부록 B. 9 개 하위 에이전트 기능 카탈로그

| # | 에이전트 | 모델 | 핵심 역할 | I/O 권한 | 주요 출력 |
|---|---|---|---|---|---|
| 1 | **Supervisor** | Haiku 4.5 | 학생 발화 → intent 분류 → 적절한 전문 에이전트로 라우팅 | 모든 학생 입력 · SessionState | route, intent, reason |
| 2 | **Pedagogy Coach** | Sonnet 4.6 | Socratic 4 단계 힌트 + mode 별 페이싱 + 자기설명 요구 | 학생 상태 · 대화 · KC · lint findings | hint, hintLevel, hintType, relatedKC, requiresSelfExplanation |
| 3 | **Code Reviewer** | Sonnet 4.6 + tools | 3 축 정적 분석 (correctness · memory safety · style) + 환각 BLOCKER 강등 | 학생 코드 · reference solution · run signals | findings[severity, line, kc, message, suggestion], topIssues |
| 4 | **Runtime Debugger** | Haiku 4.5 | 실행 결과 + 에러 메시지를 초보자 언어로 해석 + 원인 가설 2~3 개 | WASM/Judge0 실행 결과 | studentFacingMessage, hypotheses[cause, evidence, investigationQuestion, kc] |
| 5 | **Assessment** | Sonnet 4.6 + 결정론적 | 4 축 루브릭 채점 (correctness 0.5 · style 0.15 · memory 0.2 · reflection 0.15) + KC delta + Dependency Factor | 제출물 · hidden tests · code review findings · reflection | rubricScores, finalScore, kcDelta, dependencyFactor, evidence |
| 6 | **Student Modeler** | (배치 cron) | events · submissions 를 BKT 스타일로 누적 → KC 숙련도 · 오개념 갱신 · fading signal | events · submissions → mastery · misconceptions | mastery[value, confidence, observations], misconceptions[pattern, occurrences] |
| 7 | **Problem Architect** | Opus 4.7 | 새 과제 생성 + 결정적 variant 파생 (FNV-1a 해시 표절 방지) | 교사 전용 | AssignmentSeed (code, template, kcTags, rubric, hiddenTests, referenceSolution) |
| 8 | **Teacher Copilot** | Opus 4.7 | 클래스 진행 요약 + 개입 권고 (약/중/강) + 공통 오개념 집계 | 전체 학생 집계 데이터 | summary, interventionQueue[level, reasons], misconceptions, KC insights |
| 9 | **Safety Guard** | Haiku 4.5 + 결정론적 | 답 유출 차단 (Jaccard 유사도 mode 별 0.10/0.25/0.40) · PII · 프롬프트 인젝션 · 시험 모드 코드 금지 | 모든 inbound/outbound | verdict (block/allow), reasons, sanitizedPayload |

### 에이전트별 시스템 프롬프트 위치

- 정의: `.claude/agents/<name>.md` (9 개)
- 런타임: `packages/agents/src/runtime/<name>.ts`
- 도메인 스킬: `.claude/skills/{socratic-hinting, c-code-review, rubric-grading, kc-mastery-tracking, problem-templating, xapi-event, cvibe-orchestrator}/`

### 비용 최적화 전략

- **Haiku 4.5** (분류·필터·디버깅 해석): Supervisor · Safety Guard · Runtime Debugger
- **Sonnet 4.6** (일상 튜터링·리뷰·채점, 70% 토큰 효율): Pedagogy Coach · Code Reviewer · Assessment · Student Modeler
- **Opus 4.7** (저빈도 고품질): Problem Architect · Teacher Copilot
- **Prompt caching**: 모든 시스템 프롬프트 — 최대 90% 비용 절감
- **Batch API**: Student Modeler 야간 일배치 — 추가 50% 절감 (옵션)
