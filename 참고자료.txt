# 좋아 지금까지의 내용을 좀 정리해 줘. 바이브 코딩 하려고 해

## 🗒️ Answer

지금까지 논의를 **바이브 코딩 관점에서 실행 가능한 구조**로 정리하면 다음과 같습니다.

---

### 1. 핵심 개념

* **학생–AI 짝프로그래밍**

  * 학생 = Driver (주도적 코딩)
  * AI = Navigator (힌트, 설명, 리뷰)
* AI는 “답 제공자”가 아니라 **작업 수행형 에이전트**로 설계 [[3](https://www.gpters.org/research/post/0zBwFhaIKvC8R1N)]
* 대상은 대학교 1학년(처음 C언어를 배우는 학생) 
* 과제는 총 10개로 구성

---

### 2. 에이전트 목표

1. 사고 유도 (힌트 중심)
2. 코드 품질 향상
3. 학습 동기 및 몰입 강화
4. 협력적 문제 해결 경험 제공
5. 컴퓨팅 사고력 및 프로그래밍 자기효능감 향상

---

### 3. 핵심 기능

* 코드 생성(C언어) / 리팩토링
* 오류 탐지 + 원인 설명
* 단계적 힌트 제공
* 대안 비교 및 선택 지원
* 작업 흐름 이해 및 실행 (Agent 특징) [[1](https://bcho.tistory.com/1467)]

---

### 4. 화면 구조 (중요)

* 좌: 문
* 가운데: 코드 작성
* 우: AI 협력 패널(AI 챗봇)
  → “코드–AI–문제” 3축 구조

---

### 5. 인터랙션 설계

* 선택 기반: 적용 / 비교 / 무시
* 단계형 지원: 힌트 → 설명 → 코드
* 기록: 대화 로그 + 리플렉션

---

### 6. 바이브 코딩 적용 흐름

1. 문제 제시
2. 학생 코드 작성
3. AI 피드백 요청
4. 수정/개선 반복
5. 결과 공유 및 성찰

---

핵심은 단순 코딩 도구가 아니라
**“협력적 사고를 설계하는 AI 파트너 시스템”**으로 구현하는 것입니다.

# 여기서 교사는 대시보드는 어떻게?

## 🗒️ Answer

교사 대시보드는 **학생–AI 짝프로그래밍을 ‘오케스트레이션’하는 통제·지원 인터페이스**로 설계하는 것이 핵심입니다.

---

### 1. 핵심 목적

1. **학습 상태 파악** (진행률, 오류, 참여도)
2. **개입 판단 지원** (언제, 누구에게 개입할지)
3. **AI 활용 모니터링** (의존 vs 활용 균형)

→ AI는 교사를 대체하는 것이 아니라 **교사의 판단을 증강**하는 도구 [[3](https://link.springer.com/article/10.1007/s10639-024-12523-3)]

---

### 2. 필수 구성 요소

1. **학생 현황 패널**

   * 코드 진행률, 과제 완료율
   * 오류 발생 빈도 (디버깅 필요 학생 식별)

2. **AI 상호작용 로그**

   * 학생–AI 대화 내용
   * 힌트 요청 횟수 (과의존 탐지)

3. **알림/위험 신호**

   * 장시간 정체
   * 반복 오류
   * AI 답 복붙 패턴

4. **개입 도구**

   * 문제 제공
   * 힌트 직접 제공
   * 과제 난이도 조정
   * AI 개입 수준 조절 (강/중/약)

---

### 3. 고급 기능

* **학습 분석(Analytics)**: 성취도·참여도 시각화
* **AI 추천 개입**: “이 학생 개입 필요” 자동 제안
* **클래스 전체 패턴**: 공통 오류, 개념 취약점

→ AI 에이전트는 학습 데이터를 기반으로 교사 의사결정을 지원 ---

### 4. 설계 원칙

* **가시성**: 한 화면에서 전체 + 개별 상태 파악
* **개입 가능성**: 즉시 피드백 제공
* **균형 유지**: AI 자동화 vs 교사 통제

---

정리하면, 교사 대시보드는 단순 모니터링이 아니라
**“AI-학습-교사 간 상호작용을 조율하는 컨트롤 타워”**입니다.

# 부록 0. 시스템 작동 원리 (research.md §1.4)

본 시스템의 실제 작동 규칙은 6 개 원리로 명시한다. 초기 메모의 § 4 "교사 대시보드 설계 원칙" (가시성 · 개입 가능성 · 균형 유지) 은 그중 **(5) 교사 오케스트레이션 원리** 를 UI 차원으로 구체화한 것이다.

1. **학습자 주도성 원리** — 학생이 먼저 코드를 작성한 뒤 AI 가 피드백한다. AI 는 정답보다 질문과 힌트를 우선 제공하며, AI 제안을 적용하기 전 학생은 그 제안을 왜 수락하거나 거절하는지 설명해야 한다.
   - *구현 매핑*: Code-first 게이트 (AIPanel 이 학생 첫 코드 작성 전에는 정답형 응답 차단) + Accept Gate (Level 4 수락 전 자기 설명 모달 + `/api/self-explanation` 3 축 평가).

2. **단계적 지원 원리** — 지원은 `힌트 → 개념 설명 → 의사코드 → 예시 코드` 순서로만 개방한다. 시도 횟수 · 오류 유형 · 정체 시간에 따라 지원 수준을 조절하고, 초보자의 인지 부하를 줄이기 위해 피드백은 작은 단위로 분절한다.
   - *구현 매핑*: HINT_CEILING (solo 1 · pair 3 · coach 4 · exam 0) + `gating.ts` 의 attemptCount · repeatedError · stagnationSec 기반 단계 게이팅.

3. **협력적 사고 촉진 원리** — AI 는 navigator, 학생은 driver 역할을 유지한다. 에이전트는 "왜 이렇게 생각했는가?" 를 묻고, 가능한 경우 둘 이상의 해결안을 비교하게 하여 단일 정답 추종을 막는다.
   - *구현 매핑*: Pedagogy Coach 시스템 프롬프트의 "질문 우선" 규약 + 자기 설명 평가의 `transfer` 축 (대안 비교 깊이).

4. **학습 데이터 기반 개입 원리** — 오류 빈도 · 정체 시간 · 힌트 요청 수 · AI 제안 수락 패턴을 이벤트 로그로 기록한다. AI 의존도가 높은 학생은 `intervention flag` 를 생성해 교사 개입 큐에 올린다.
   - *구현 매핑*: xAPI events 영구 저장 + `learning-signals.ts` + Dependency Factor 0~1 (낙인 방지 — 점수 감점 X, 교사 전용) + `/queue` 페이지의 research.md §6.4 트리거.

5. **교사 오케스트레이션 원리** — 교사는 전체 진행률과 개별 상태를 동시에 확인하고 AI 개입 수준을 약 · 중 · 강으로 조정한다. 공통 오류와 취약 개념은 다음 수업의 즉시 피드백 소재로 환류한다.
   - *구현 매핑*: 교사 개요 7-layer 코크핏 (Health · At-Risk · Today · Trend · Action · LiveEvents · ModeDonut) + InterventionActions (모드 강제 · 쪽지 · 잠금 해제) + Common Misconceptions 집계.

6. **반복 개선 원리** — `수업 적용 → 로그 분석 → 설계 수정` 의 순환을 운영 단위로 고정한다. 프롬프트 · UI · 피드백 규칙 · 에이전트 조합 패턴은 버전 관리되며 재사용 가능한 설계 자산으로 축적한다.
   - *구현 매핑*: CLAUDE.md 변경 이력 누적 + 운영 플레이북 (과제 교체 8 단계) + Catalog Drift cron + Promptfoo 회귀 테스트 + Langfuse trace.

# 부록 B. 9 개 하위 에이전트 기능 카탈로그

| # | 에이전트 | 모델 | 핵심 역할 | I/O 권한 | 주요 출력 |
|---|---|---|---|---|---|
| 1 | **Supervisor** | Haiku 4.5 | 학생 발화 → intent 분류 → 적절한 전문 에이전트로 라우팅 | 모든 학생 입력 · SessionState | route, intent, reason |
| 2 | **Pedagogy Coach** | Sonnet 4.6 | Socratic 4 단계 힌트 + mode 별 페이싱 + 자기설명 요구 | 학생 상태 · 대화 · KC · lint findings | hint, hintLevel, hintType, relatedKC, requiresSelfExplanation |
| 3 | **Code Reviewer** | Sonnet 4.6 + tools | 3 축 정적 분석 (correctness · memory safety · style) + 환각 BLOCKER 강등 | 학생 코드 · reference solution · run signals | findings[severity, line, kc, message, suggestion], topIssues |
| 4 | **Runtime Debugger** | Haiku 4.5 | 실행 결과 + 에러 메시지를 초보자 언어로 해석 + 원인 가설 2~3 개 | WASM/Judge0 실행 결과 | studentFacingMessage, hypotheses[cause, evidence, investigationQuestion, kc] |
| 5 | **Assessment** | Sonnet 4.6 + 결정론적 | 4 축 루브릭 채점 (correctness 0.5 · style 0.15 · memory 0.2 · reflection 0.15) + KC delta + Dependency Factor | 제출물 · hidden tests · code review findings · reflection | rubricScores, finalScore, kcDelta, dependencyFactor, evidence |
| 6 | **Student Modeler** | (배치 cron) | events · submissions 를 BKT 스타일로 누적 → KC 숙련도 · 오개념 갱신 · fading signal | events · submissions → mastery · misconceptions | mastery[value, confidence, observations], misconceptions[pattern, occurrences] |
| 7 | **Problem Architect** | Opus 4.7 | 새 과제 생성 + 결정적 variant 파생 (FNV-1a 해시 표절 방지) | 교사 전용 | AssignmentSeed (code, template, kcTags, rubric, hiddenTests, referenceSolution) |
| 8 | **Teacher Copilot** | Opus 4.7 | 클래스 진행 요약 + 개입 권고 (약/중/강) + 공통 오개념 집계 | 전체 학생 집계 데이터 | summary, interventionQueue[level, reasons], misconceptions, KC insights |
| 9 | **Safety Guard** | Haiku 4.5 + 결정론적 | 답 유출 차단 (Jaccard 유사도 mode 별 0.10/0.25/0.40) · PII · 프롬프트 인젝션 · 시험 모드 코드 금지 | 모든 inbound/outbound | verdict (block/allow), reasons, sanitizedPayload |

# 부록 C. 에이전트 설계원리 (2026-05-15)

본 멀티 에이전트 시스템의 모든 구현 결정 (4단계 힌트 게이팅·HINT_CEILING·자기설명 모달·5단계 Proficiency 등) 은 다음 7대 설계원리로 환원·정당화된다. 9개 에이전트의 시스템 프롬프트·게이팅 규칙·라우팅 정책은 이 원리들로 점검·검증한다.

## 1. 학습자 주도 문제해결 원리

- 학생이 첫 코드를 작성하기 전에는 AI가 완성 코드나 직접 정답을 제공하지 않도록 한다.
- 학생이 과제의 요구 조건, 입력, 처리 과정, 출력 형식을 먼저 파악하도록 안내한다.
- AI 피드백은 학생이 작성한 코드와 시도 이력을 바탕으로 제공한다.
- AI는 정답 제시보다 질문, 점검, 방향 힌트를 우선 제공한다.

→ 구현 매핑: Pedagogy Coach 4단계 게이팅 · Safety Guard reference_solution Jaccard 유사도 차단 · 첫 코드 작성 전 힌트 레벨 1 잠금.

## 2. 단계적 스캐폴딩 원리

- 첫 번째 지원은 문제 해결 방향이나 확인할 조건을 알려주는 힌트 수준으로 제한한다.
- 학생의 반복 시도, 동일 오류, 장시간 정체가 확인될 때 더 구체적인 개념 설명이나 절차 안내를 제공한다.
- 지원은 힌트 → 개념 설명 → 절차 안내 → 의사코드 → 예시 코드의 순서로 확장한다.
- 예시 코드는 충분한 시도와 자기설명 이후에만 제한적으로 제공한다.
- 피드백은 한 번에 여러 내용을 제시하지 않고 하나의 핵심 오류나 개념 단위로 분절한다.

→ 구현 매핑: Pedagogy Coach Level 1~4 (정의 질문 → 개념 → 의사코드 → 예시) · HINT_CEILING (solo=1 · pair=3 · coach=4) · Code Reviewer findings 1건씩 분절 제시.

## 3. 오류 기반 피드백 원리

- 컴파일 오류, 실행 오류, 논리 오류, 테스트 실패를 구분하여 피드백한다.
- 컴파일러 메시지를 그대로 제시하는 데 그치지 않고 학생 수준에 맞게 풀어 설명한다.
- 오류 위치뿐 아니라 오류가 발생한 이유와 확인해야 할 변수, 조건식, 반복 구조를 함께 안내한다.
- 정답 코드를 바로 제시하기보다 학생이 직접 수정할 수 있는 점검 질문을 제공한다.
- 동일 오류가 반복될 경우 관련 개념을 다시 확인하도록 안내한다.

→ 구현 매핑: Runtime Debugger 원인 가설 2~3개 제시 · Code Reviewer 카테고리 분리 (correctness · memory-safety · style) · Assessment misconception 누적 추적.

## 4. 자기설명 기반 AI 수용 원리

- AI의 수정 제안, 힌트, 코드 조각을 적용하기 전 학생이 왜 이를 수락하려는지 설명하게 한다.
- AI 제안을 거절하는 경우에도 왜 적절하지 않다고 판단했는지 작성하게 한다.
- 자기설명은 수정 내용, 수정 이유, 예상 결과를 중심으로 간단히 작성하도록 한다.
- 학생이 어느 코드 부분을 왜 바꾸는지 명시하도록 유도한다.
- 자기설명이 부족할 경우 AI가 추가 질문을 통해 설명을 보완하게 한다.

→ 구현 매핑: AIPanel Level 4 수락 전 자기설명 모달 · `/api/self-explanation` 3축(specificity·causality·transfer) Claude 평가 · `Verbs.selfExplanationSubmitted` xAPI 기록.

## 5. 자기조절학습 지원 원리

- 학생이 현재 과제의 어느 단계에 있는지 진행 상태를 시각적으로 확인할 수 있게 한다.
- 전체 과제를 작은 목표로 나누어 단계별로 수행하도록 안내한다.
- 짧은 집중 학습 시간을 설정하여 몰입을 유지하도록 지원한다.
- 장시간 코드 변화 없음, 반복 실행 실패, 힌트 과다 요청과 같은 막힘 신호를 감지한다.
- 막힘이 감지되면 변수 값 확인, 작은 입력값 테스트, 조건식 점검, 이전 단계로 돌아가기와 같은 구체적인 회복 행동을 제안한다.

→ 구현 매핑: ProgressTracker 좌측 막대 · FocusMode 15분 타이머 · StuckDiagnostic 4선택지 패널 · WalkthroughPrompt 30분 정체 트리거 · AssignmentPanel 학습목표 표시.

## 6. 정서적 안정 지원 원리

- AI 피드백은 "틀렸습니다", "잘못했습니다"와 같은 평가적 표현보다 "현재 코드에서 함께 확인해 볼 부분이 있습니다", "이 오류는 많은 초보자가 자주 만나는 문제입니다"와 같은 안정적 표현을 사용한다.
- 반복 실패가 발생했을 때 학생의 능력 부족으로 해석하지 않고, 점검할 코드 요소와 다음 시도 단계를 구체적으로 제시한다.
- 학생이 도움을 요청했을 때 부끄러움이나 실패감을 느끼지 않도록, 도움 요청을 학습 전략의 일부로 긍정적으로 안내한다.
- 오류 메시지, 테스트 실패, 낮은 진행률 등은 학생에게 위협적으로 제시하지 않고, 개선 가능한 학습 신호로 표현한다.
- 학생의 수행 수준은 점수나 등수처럼 경쟁적·평가적으로 제시하지 않고, '탁월', '능숙', '성장 중', '도움 필요'와 같은 서술형 수준 표현으로 제시한다.

→ 구현 매핑: 5단계 Proficiency 배지 (🏆 탁월 / ✨ 능숙 / 📘 발전 중 / 🌱 시작 / 🛠 진행) · 시도 delta "↑8p" 자기 과거 비교만 · raw score 학생 화면 노출 금지 · Microcelebration 토스트 · 시스템 프롬프트의 평가적 표현 금지 규약.
